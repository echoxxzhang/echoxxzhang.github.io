Mysql数据库Day1-关系型数据库

Posted on 2019-08-01 | | 阅读数

前言

目前市场上比较流行的数据库分为两大类：关系型数据库（Mysql）和非关系型数据库（MongoDB、Redis）

我打算在数据库系列先介绍Mysql数据库，以后看情况更新 MongoDB、Redis（其实还没学，哈哈）


关系型数据库

关系模型的定义

关系模型使用的存储结构是多个二维表，即反映事物及其联系的数据描述是以平面表格形式体现的（类似于Excel表格）在关系模型中，记录之间的联系时通过不同关系中同名属性来体现的。



关系模型需要满足的条件如下：

  1. 数据表通常是一个由行和列组成的二维组，它说明数据库中对象的属性
  2. 数据表的行通常叫做记录或元组，它代表众多具有相同属性的对象的一个
  3. 数据表中的列通常叫做字段或属性，它代表相应数据库中存储对象的共有的属性
  4. 主键和外键，数据表之间的关联实际上是通过键（key）来实现的，所谓“键”是指数据表的一个字段。键分为主键（Primary KeyForeign Key）两种，它们都在数据表连接的过程中起着重要的作用。
  • 主键: 是数据表中具有唯一性的字段，也就是说数据表中任意两条记录都不可能拥有相同的主键字段。
  • 外键: 一个数据表将使用该数据表中的外键连接到其他数据表，而这个外键字段在其他数据表中将作为主键字段出现。
  1. 关系表必须符合的条件：
  • 信息原则：存储在单元中的数据必须是原始的，每个单元只能存储一条数据
  • 存储在列下的数据必须具有相同数据类型；列没有顺序，但有一个唯一性的名称
  • 行与行之间是独立且无序的
  • 实体完整性原则（主键保证）和引用完整性原则（外键），都不能为空


三范式理论

设计Mysql 数据库时，考虑到一些规范，于是提出了“三范式理论”，他们分别是：

  • 第一范式：列不可拆分，这个存储的列，不能再拆了（基于所描述实体的数据库的具体情况）

  • 第二范式：唯一标识，需要有一个属性可以唯一的找到一个对象，也就是说确保表中的每列都和主键相关

  • 第三范式：引用主键，确保数据表中的每一列数据都和主键直接相关，而不能间接相关。



注意事项：

  1. 第二范式与第三范式的本质区别：在于有没有分出两张表。
  2. 第二范式是说一张表中包含了多种不同实体的属性，那么必须要分成多张表，第三范式是要求已经分好了多张表的话，一张表中只能有另一张标的ID，而不能有其他任何信息（其他任何信息，一律用主键在另一张表中查询）。
  3. 必须先满足第一范式才能满足第二范式，必须同时满足第一第二范式才能满足第三范式。

Mysql基础

理论知识

  • 一个数据库就是一个完整的业务单元，可以包含多张表，数据被存储在表中。

  • 在表中为了更加准确的存数据，保证数据的正确有效，可以在创建表的时候，为表添加一些强制性的验证 ，包括数据段的类型、约束。



结构化查询语言

  • SQL是一种专门用来与数据库通信的标准语言，SQL可以进行的工作是：数据查询、数据操纵、数据定义和数据控制。所有关系型（目前大多数数据库都是关系型数据库,基于E-R模型）都支持SQL语言。

  • 在SQL中，以分号进行语句的分割（每一条SQL语句都需要；） 而SQL 对于大小写、空格、换行符不敏感。

  • 但是SQL的字符对半角、全角是敏感的，注意我们的输入法应该是英文格式，不然会报错
